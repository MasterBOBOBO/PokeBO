第 0 章・導論

什麼是 OpenAI Codex

什麼是 Codex，為什麼它很重要

Codex 是 OpenAI 的自主軟體工程代理（software engineering agent）。它不是自動補全工具：你給它一個完整的工作單位——「修好結帳流程的失敗測試」、「幫訂單 API 加上分頁」、「把整個專案升級到 React 19」——它會讀懂整個程式碼庫、跨多個檔案撰寫與修改程式、在隔離沙箱中執行測試，最後開出 Pull Request 給你審核。

核心重點：Codex 是「一個代理、三種介面」。同一個代理可以在你的終端機（開源的 Codex CLI）、你的編輯器（VS Code 等 IDE 擴充功能），以及雲端（透過 ChatGPT 網頁、iOS App，甚至 Slack）工作。本機介面適合即時協作，雲端介面則能在背景平行執行多個任務——你下班了它還在寫。

它重要的原因和所有前緣程式代理相同：任務越大、越完整，相對於傳統工具的優勢就越明顯。補全工具幫你省下打字時間；Codex 省下的是「整個任務」的時間。

從補全模型到工程代理：你該知道的演進史

  • 2021 年：初代 Codex 模型問世，是 GitHub Copilot 背後的補全引擎——當時它只是「幫你接下一行」。
  • 2025 年 4～5 月：Codex 以全新面貌回歸：開源的 Codex CLI 上線，接著雲端代理版在 ChatGPT 中推出研究預覽。從「補全」正式轉向「代理」。
  • 2025 年下半年：專用模型 GPT-5-Codex、GPT-5.1-Codex 系列陸續登場，VS Code 擴充功能安裝量衝破數百萬。
  • 2026 年 2 月 24 日：GPT-5.3-Codex 發布，速度提升 25%，在 SWE-Bench Pro 與 Terminal-Bench 2.0 創下新紀錄，且比先前任何模型都省 token。
  • 2026 年上半年：OpenAI 將旗艦整併——GPT-5.5 成為 Codex 的建議模型（不再出專用的 5.5-codex 變體），GPT-5.3-Codex 進入退役程序。6 月起 Codex 也登上 Amazon Bedrock。

這段歷史有一個實務意義：Codex 的模型與方案更新非常快，本指南的原則性內容（工作方法、提示詞、整合方式）壽命很長，但具體型號與價格請以官方頁面為準。

本指南適合誰

  • 職業開發者——想從「自己寫每一行」升級成「委派整個任務」：修 Bug、加功能、升級框架、補測試，一次交付一整包
  • 技術主管與團隊——想把程式碼審查、CI 失敗分流、重複性維護工作自動化，讓團隊專注在設計決策上
  • 獨立開發者與創業者——想用最小的人力打造真正上線的產品，從 MVP 到營運自動化
  • 非工程背景但想做出東西的人——Codex 的雲端介面讓你用一段描述就能得到可運行的原型，不用碰終端機

一個簡單的例子幫你定位：如果你的需求是「這行程式什麼意思」，任何聊天模型 30 秒就能回答，用不到 Codex。但如果你的需求是「這個 legacy 專案沒人敢動，改一行要測三天」——那正是這個代理存在的理由。

模組 1・基礎

基礎知識

你需要理解的 6 個概念

1. 代理，不是補全。Codex 接的是「工作單位」：它自己規劃步驟、修改多個檔案、跑測試、驗證結果。你的角色從「打字的人」變成「派工和驗收的人」。

2. 三種介面，一個代理。CLI（終端機，開源）適合本機即時協作與腳本化；IDE 擴充功能（VS Code 等）適合邊看邊改；雲端任務（ChatGPT 網頁／iOS／Slack）適合「丟出去、背景跑、回來收 PR」。三者共用你的設定與習慣。

3. AGENTS.md——給代理看的 README。放在專案根目錄的說明檔，告訴 Codex 怎麼建置、怎麼跑測試、遵守什麼規範。官方經驗法則：保持精簡（約 150 行內，上限 32 KiB）。檢驗標準很簡單：任何人打開 Codex 說「跑測試」，第一次就能成功——做不到就代表 AGENTS.md 缺了必要指令。

4. 沙箱與許可模式。Codex 在隔離沙箱中執行指令，網路存取與檔案寫入權限由你設定。從「每步確認」到「全自動」有多檔可調，讓你依任務風險決定放多少手。

5. 平行雲端任務。雲端版可同時發動多個任務，各自在獨立容器中連上你的 GitHub 儲存庫作業，完成後開 PR。進階使用者的日常：早上派 4 張工單，下午收 4 個 PR 逐一驗收。

6. 專用模型的世代交替。Codex 歷代由專用模型驅動（GPT-5-Codex → 5.1 → 5.3-Codex），2026 年中起 OpenAI 建議直接使用旗艦通用模型 GPT-5.5。對使用者的意義：選模型時看官方目前的預設建議即可，不必追型號。

數據解讀的重點不是排名，而是趨勢：Codex 系模型的強項是「工程流程的完整執行」——終端機操作、跑測試、開 PR 這一整條路，而且 token 效率一代比一代好。

⚠️ 用量管理的注意事項

訂閱方案的 Codex 用量按「任務量與複雜度」計，不是無限吃到飽。長時間代理任務（大型重構、整包遷移）消耗得快；用完可等額度重置或加購 credits。先發動價值最高的任務，例行小事留給便宜的模型或補全工具。

🔒 安全與隱私注意事項

雲端任務會在 OpenAI 的容器中存取你授權的 GitHub 儲存庫——企業導入前先確認資料處理政策與訓練資料退出（opt-out）設定。本機 CLI 的沙箱預設限制網路與寫入範圍，調成全自動模式前，想清楚這個專案值不值得。

所有案例中不斷重複的模式

成功導入的團隊都做了同一件事：先把專案整理到「代理能自己驗證成果」的狀態——測試能跑、AGENTS.md 寫清楚、CI 是綠的——然後才開始委派。失敗的導入幾乎都是反過來：專案一團亂就想靠代理救。

這給我們的啟示與所有前緣代理相同：Codex 獎勵的不是會下指令的人，而是能提供「可驗證環境」的人。測試齊全的專案，代理能自己確認做對了；沒有測試的專案，你付出的驗收時間會吃掉省下的開發時間。

類別 A：理解程式碼

提示詞 1：陌生專案快速上手

我剛接手這個專案。請完整探索程式碼庫後給我一份上手報告：1）整體架構與主要模組的職責；2）資料流：一個請求從進入到回應經過哪些層；3）建置、測試、部署的實際指令（親自執行驗證過再寫）；4）你發現的 3 個最可能踩雷的地方。不要修改任何檔案。

用途：把接手專案的摸索期從數天縮短到一小時。預期結果：一份經過實際驗證的上手文件。

提示詞 2：技術債盤點與優先排序

請盤點這個程式碼庫的技術債：重複的邏輯、過時的依賴、沒有測試保護的關鍵路徑、被註解掉的死程式碼。輸出一張表：問題、位置、風險等級（高／中／低）、預估修復規模、建議順序。判斷依據要引用具體檔案與行號。先不要修，我要先跟團隊排優先順序。

用途：把「感覺很亂」變成可排程的工作清單。預期結果：有依據、可追溯的技術債清單。

提示詞 3：跨檔案行為追蹤

請追蹤［某功能，例如：使用者改密碼］在這個程式碼庫的完整路徑：入口在哪、經過哪些函式與服務、寫入哪些資料表、觸發哪些副作用（郵件、快取、事件）。用一張流程清單呈現，每一步附檔案與行號。最後指出這條路徑上沒有被任何測試覆蓋的環節。

用途：改動前先看清全貌，避免改一處壞三處。預期結果：附行號的完整行為地圖。

提示詞 4：架構決策評估

我們考慮把［現況，例如：單體架構的通知模組］改成［目標，例如：獨立服務］。請基於這個程式碼庫的實際狀況（不是通論）評估：牽動哪些模組、資料遷移的難點、可以分幾個階段安全推進、每階段的回滾方案。結論給我一個明確建議：做、不做、或先做哪一小步。

用途：用真實程式碼支撐架構決策，不靠白板空想。預期結果：基於實際依賴關係的分階段方案。

類別 B：修復與重構

提示詞 5：工單式 Bug 修復

Bug：［一句話描述］。重現步驟：［步驟］。預期行為：［描述］。實際行為：［描述］。請先寫一個能重現此 bug 的失敗測試，再修復它讓測試轉綠，最後跑完整測試套件確認沒有回歸。PR 說明中要包含：根本原因、為什麼這樣修、還有哪些地方可能有同類問題。

用途：標準的「測試先行」修復流程，成果可驗證。預期結果：附失敗測試轉綠證明的修復 PR。

提示詞 6：安全重構

請重構［目標檔案／模組］：消除重複、拆分過長的函式、改善命名，但外部行為必須完全不變。規則：每一步重構前先確認現有測試通過；若關鍵路徑沒有測試，先補測試再動手；分成多個小 commit，每個 commit 訊息說明該步做了什麼。最後給我重構前後的對照摘要。

用途：把「不敢動的程式碼」變成可維護的程式碼。預期結果：行為不變、測試護航的重構 PR。

提示詞 7：依賴與框架升級

請把這個專案從［目前版本，例如：React 18］升級到［目標版本，例如：React 19］。流程：1）列出所有破壞性變更與受影響檔案；2）逐批修改，每批之後跑建置與測試；3）棄用 API 一併遷移到新寫法；4）交付一份升級記錄：改了什麼、哪裡曾壞掉、怎麼解的。過程中不要改動與升級無關的程式碼。

用途：Codex 優勢最大的任務類型之一——大範圍、機械性但需要判斷。預期結果：升級完成、測試通過、附完整記錄。

提示詞 8：效能診斷與優化

［某功能／端點］太慢。請先量測再優化：1）加上計時或使用分析工具找出真正的瓶頸，附數據；2）針對前 2 個瓶頸提出修法並實作；3）優化後重新量測，給我前後對比數字；4）明確說明有哪些取捨（記憶體、可讀性、快取失效風險）。沒有數據支持的優化不要做。

用途：強制「先量測、後優化」，避免憑感覺調校。預期結果：有前後數據對比的優化 PR。

類別 C：建造新功能

提示詞 9：完整功能開發（工單式）

功能需求：［描述使用者故事：作為…我想要…以便…］。驗收標準：［列出 3～5 條可檢驗的條件］。請依這個程式碼庫的既有模式實作（先觀察類似功能怎麼寫），包含：後端邏輯、前端介面、錯誤處理、測試。每個驗收標準都要有對應的測試。完成後在 PR 中逐條對照驗收標準說明。

用途：一張標準工單直接變成一個可驗收的 PR。預期結果：符合專案風格、驗收標準逐條達成的功能。

提示詞 10：一段話生出可用原型

請建造［用 2～3 句話描述工具／App：給誰用、解決什麼問題］。技術選型、架構與 UI 由你決定，但要選最簡單能達成目標的方案。我要的是能直接執行、用起來舒服的成品，不是骨架。請附上啟動指令，並在交付前自己實際跑過主要流程確認可用。

用途：用最快速度驗證點子，適合雲端任務背景執行。預期結果：可直接啟動試用的原型。

提示詞 11：API 設計與實作

請為［資源／功能］設計並實作 REST API：1）先給我端點設計（路徑、方法、請求／回應格式、錯誤碼），等我確認；2）確認後實作，含輸入驗證、權限檢查、分頁；3）每個端點附整合測試；4）產出一份可直接給前端同事看的 API 文件。遵循這個專案既有的 API 慣例。

用途：設計先行、確認後實作的兩段式流程。預期結果：文件、實作、測試三位一體的 API。

提示詞 12：資料處理流程建置

我需要一個資料流程：從［來源，例如：每日的 CSV 報表］到［目的，例如：清理後寫入資料庫並產生摘要通知］。請建造：擷取、驗證（壞資料要隔離不要中斷）、轉換、載入、執行記錄。用一份真實樣本資料測試整條流程，交付時附上執行方式與失敗時的排查指南。

用途：把重複的資料雜務變成一條可靠的管線。預期結果：經樣本資料驗證的完整 ETL 流程。

類別 D：品質與審查

提示詞 13：測試覆蓋率衝刺

請提升［模組／目錄］的測試覆蓋率：1）先跑覆蓋率工具，找出未覆蓋的關鍵路徑（商業邏輯優先，getter/setter 跳過）；2）為前 10 個最重要的缺口補上測試，包含正常路徑、邊界條件、錯誤處理；3）測試要能抓到真實的錯誤——寫完後試著在原始碼植入一個小錯，確認測試會失敗。交付前後覆蓋率數字。

用途：補的是「有效測試」，不是灌覆蓋率的假測試。預期結果：覆蓋率提升＋測試有效性驗證。

提示詞 14：深度 PR 審查

請審查這個 PR（或分支 diff）：1）邏輯錯誤與未處理的邊界條件（最優先）；2）與程式碼庫既有慣例的不一致；3）效能與安全疑慮；4）測試是否真的覆蓋了變更。每個發現附：位置、嚴重程度、具體修改建議。最後給一個明確結論：可合併／小修後合併／需要重做，以及理由。

用途：本機版的深度審查；GitHub 整合版可自動跑（見模組 5）。預期結果：有嚴重度分級與明確結論的審查報告。

提示詞 15：安全健檢

請對這個程式碼庫做一次防禦性安全健檢：輸入驗證缺口、注入風險、硬編碼的金鑰或密碼、過時且有已知漏洞的依賴、權限檢查遺漏。輸出：問題清單（位置＋風險說明＋修法建議），依嚴重度排序。金鑰類問題直接修（移入環境變數），其他先報告，由我決定修哪些。

用途：上線前的例行防禦性檢查。預期結果：分級的安全問題清單，高危項目已處理。

提示詞 16：文件同步更新

這個專案的文件（README、API 文件、註解）已經和程式碼脫節。請：1）逐一核對文件描述與實際程式碼行為，列出所有不一致；2）以「程式碼是事實」為原則更新文件；3）安裝與啟動步驟要在乾淨環境親自跑過一遍再寫；4）刪掉描述已不存在功能的段落。交付一份「文件修正對照表」。

用途：解決「文件永遠是舊的」這個千年問題。預期結果：與程式碼一致、經實測的文件。

類別 E：自動化交付

提示詞 17：CI 失敗分流

CI 掛了，這是失敗記錄：［貼上或附連結］。請：1）判斷失敗類型：真實的程式錯誤、不穩定測試（flaky）、環境問題；2）如果是程式錯誤，定位根因並修復；3）如果是 flaky 測試，找出不穩定的原因（時序、共享狀態、外部依賴）並根治，不要只加重試；4）說明你的判斷依據。目標是讓這條 pipeline 穩定轉綠。

用途：CI 紅燈的標準處理流程，可搭配 codex exec 自動化。預期結果：轉綠的 pipeline 與根因說明。

提示詞 18：AGENTS.md 建立或健檢

請為這個專案［建立／檢修］AGENTS.md：1）親自執行建置、測試、lint 指令，確認寫進去的每一條都真的能跑；2）記錄專案的程式風格慣例（從現有程式碼歸納，不要發明新規則）；3）標注絕對不能動的區域（如自動生成的檔案）；4）保持精簡：目標 150 行內。完成後的檢驗標準：任何人對你說「跑測試」，你第一次就能成功。

用途：AGENTS.md 是所有委派的地基，這個提示詞讓代理自己打地基。預期結果：精簡、每條指令都驗證過的 AGENTS.md。

提示詞 19：批次派工計畫

這是我的待辦清單：［貼上工單或任務列表］。請幫我規劃平行派工：1）哪些任務彼此獨立、可以同時跑（不會改到同一批檔案）；2）哪些有依賴、必須排序；3）把每個任務改寫成完整的工單式提示詞（背景、範圍、驗收標準）；4）標注每個任務建議用本機還是雲端跑。我要直接照這份計畫發任務。

用途：把「平行雲端任務」的產能真正用滿。預期結果：可直接執行的派工計畫與現成提示詞。

提示詞 20：上線前總檢查

這個專案準備上線。請做最後總檢查並輸出報告：1）完整測試套件與建置是否通過（實際執行）；2）提示詞 15 的安全健檢快速版；3）設定檔中是否還有開發環境殘留（debug 開關、測試金鑰、localhost）；4）錯誤發生時使用者會看到什麼（實際觸發一個錯誤看看）。每項給通過／不通過，不通過的直接修，改動列入報告。

用途：發布前的守門員，抓的是「上線才會爆」的問題。預期結果：一份可留檔的上線檢查報告。

模組 4・進階

15 個實戰技巧

心態篇：把它當同事，不當工具

1. 派「工作單位」，不派「程式問題」。官方最佳實踐的第一條：Codex 接完整任務的表現遠勝零碎問答。「修好這張工單」優於「這個函式怎麼寫」。

2. 把它當一個「需要入職訓練的同事」。官方說法：與其當一次性助手，不如當一個你會持續設定與改進的隊友。AGENTS.md 是它的入職手冊，skills 是它的標準作業程序，每次糾正都寫回去。

3. 平行派工，分批驗收。雲端任務可以同時跑好幾個。進階使用者早上派 3～4 張互不相干的工單，下午統一收 PR。你的瓶頸會從「寫程式的速度」變成「驗收的速度」。

4. 低風險任務先行，逐步放權。成功導入的標準路徑：文件更新 → 測試補強 → Bug 修復 → 功能開發。每一級建立信任後才升下一級，許可模式也跟著逐步放寬。

5. 把心力投資在「工單品質」和「驗收流程」。模型能力是天花板，你的工單品質是地板。地板太低,天花板再高也沒用。

設定篇：AGENTS.md 與環境

6. AGENTS.md 保持精簡。官方經驗法則約 150 行（實際上限 32 KiB）。寫太長代理反而抓不到重點。只放三類東西：建置測試指令、專案慣例、禁區。

7. 用「跑測試」測試你的 AGENTS.md。檢驗標準：任何人打開 Codex 說「run the tests」，第一次就成功。失敗就代表缺了必要的設定指令——這是官方文件明講的判斷法。

8. 把重複的工作變成 skills。同一種任務派過三次（如「按我們的規範出報表」），就把流程沉淀成可重用的技能檔，之後一句話喚起。

9. 團隊共用審查規範。把 code_review.md 寫好、從 AGENTS.md 引用，Codex 審 PR 時就會遵循——整個團隊、所有儲存庫的審查標準就此一致。

10. 用 MCP 接上外部系統。需要查工單系統、內部文件、資料庫時，接上對應的 MCP 伺服器，讓代理自己取用上下文，而不是你手動貼上。

執行與成本篇

11. 依風險調許可模式。陌生專案用「每步確認」，測試齊全的專案用「自動編輯、確認執行」，一次性沙箱實驗才開全自動。全自動模式配上沒測試的專案是災難配方。

12. 大任務切成可驗收的段落。「整包遷移」聽起來很帥，但一個 200 檔案的 PR 沒人審得動。要求代理分階段交付，每段一個可獨立合併的 PR。

13. 用量花在刀口上。訂閱額度按任務量與複雜度消耗。例行補全交給便宜工具，Codex 留給「一張工單以上」規模的事。額度吃緊時優先跑價值最高的任務。

14. 讓 CI 幫你把關，而不是你自己。代理開的 PR 一律過完整 CI（測試、lint、型別檢查）。機器能擋掉的問題不要用人眼擋。

15. 要求「證據式回報」。永遠加上：「交付時附上測試輸出、實際執行結果，沒驗證過的部分明講。」代理有自我驗證能力，這句話強制它展示證據。

不斷重演的錯誤（以及如何避免）

  • 把它當聊天機器人：問「這段程式什麼意思」用一般模型就好，Codex 的價值在完整任務
  • 在沒有測試的專案上全自動放行：代理無法自我驗證，你也無法快速驗收，雙輸
  • AGENTS.md 寫成百科全書：超過重點容量，代理反而忽略關鍵規則
  • 一次派出巨型任務：不可驗收的 PR 等於沒有 PR
  • 糾正完不沉淀：同一個錯誤糾正三次還沒寫進 AGENTS.md，是你的問題不是它的

模組 5・進階

強大的工具組合

Codex + GitHub（自動程式碼審查）

開啟 GitHub 整合後，Codex 可以在每個 PR 合併前自動審查：標記邏輯錯誤與未處理的邊界條件、提出具體修法。也可以被動觸發——在 PR 留言 @Codex 加指示即可。

實務流程：先用「僅標記、不阻擋」模式跑兩週建立信任，再把高嚴重度發現設為合併門檻。搭配技巧 9 的 code_review.md 統一標準。

可衡量收益：人類審查者只看代理標過的重點，審查週期明顯縮短。

Codex + Slack（討論串直接變任務）

在頻道或討論串中 @Codex 加上需求描述，它會建立雲端任務並把結果回覆到原討論串。Bug 回報、需求討論到修復 PR 之間的距離變成一則訊息。

實務流程：在工程頻道建立慣例：可直接委派的訊息用固定格式（現象＋重現步驟＋預期行為），讓任何人都能一鍵派工。

可衡量收益：「回報到修復」的流程不再經過工單系統的排隊。

Codex + CI/CD（codex exec 自動化）

CLI 的 codex exec 可以在腳本與 pipeline 中非互動執行，這讓穩定的工作流程可以完全自動化：夜間跑安全掃描、CI 紅燈自動分流（提示詞 17）、每週自動出依賴更新 PR。

實務流程：挑一個你已經手動跑過多次、結果穩定的流程（先手動、後自動是官方建議的順序），把它包進排程任務。

可衡量收益：例行維護從「有人記得才做」變成「排程保證做」。

Codex + MCP（接上你的內部系統）

MCP 讓 Codex 取用外部上下文：工單系統、內部文件、資料庫、監控面板。代理自己查資料，而不是等你貼上。

實務流程：從一個最常查的系統開始（通常是工單系統），設定完成後在 AGENTS.md 註明何時該用哪個工具。

可衡量收益：派工時不用再手動整理上下文，工單編號就是完整輸入。

Codex + IDE（VS Code 擴充功能）

近千萬安裝量的主流介面。適合「邊看邊改」的工作型態：你看得到每個 diff、隨時介入，和雲端任務的「全委派」互補。

實務流程：白天在 IDE 處理需要即時判斷的工作，需要長時間跑的任務用「移交到雲端」功能丟出去背景執行。

可衡量收益：本機與雲端無縫接力，工作不因下班中斷。

Codex + 行動裝置（iOS App）

在手機上派工、看進度、審 PR 摘要。適合通勤時把今天的任務排出去，到公司時已經有初步結果。

實務流程：搭配提示詞 19 的批次派工計畫，把「派工」變成早晨例行事務。

模組 6・變現

如何變現賺錢

收入區間為市場常見行情的參考值（以美元計），實際依地區、定位與作品集而異。

方法 1：成品開發服務

Codex 讓你賣「交付物」而不是「工時」：一個上線的網站、一個修好的模組、一次完整的框架升級、一套補齊的測試。

潛在收入：每個專案 $500～$5,000。每月 3～4 個專案：$1,500～$20,000。

如何開始：挑一種你能驗收品質的交付物（這是關鍵——你賣的是你的驗收能力），做 2 個範例當作品集，標固定價格。

方法 2：微型 SaaS 與利基工具

用提示詞 10 把點子變成原型、提示詞 9 把原型變成產品、CI 整合讓維護成本趨近於零——單人維護多個小產品從此可行。

潛在收入：利基工具月費 $9～$29。50 位付費使用者：每個工具每月 $450～$1,450 經常性收入。

如何開始：找你社群裡重複出現的痛點，一週內做出最小版本，先賣給 10 個人再打磨。

方法 3：程式碼健檢與救援服務

提示詞 1、2、15 的組合就是一個可販售的服務：接手評估、技術債盤點、安全健檢、「沒人敢動的專案」救援。市場上到處都是接手了外包爛尾專案的公司。

潛在收入：健檢報告 $800～$3,000；救援專案 $3,000～$15,000。

如何開始：用開源專案做一份示範健檢報告，讓潛在客戶看到交付物長什麼樣。

方法 4：AI 開發流程培訓與顧問

大多數團隊買了訂閱卻只把 Codex 當高級補全用。教團隊「工單式委派、AGENTS.md 設定、審查自動化」是現在需求最高的顧問服務。

潛在收入：工作坊 $500～$2,000／場；月度顧問每客戶 $1,000～$5,000。

如何開始：把模組 3～5 內化成一場 45 分鐘的實演，免費講給 2 家公司聽，換取見證與轉介。

方法 5：自動化維護代理商

結合模組 5：幫客戶架好 GitHub 審查、CI 自動化、Slack 派工，然後收月費維護。企業願意為「持續運轉的工程流程」付經常性費用。

潛在收入：建置專案 $2,000～$8,000，另加每客戶每月 $200～$800 維護費。8 個維護客戶：每月 $1,600～$6,400 經常性基本盤。

如何開始：先把自己的專案全面自動化，用前後數據當銷售證明。

模組 7・進階

3 個進階應用案例

案例 1：舊系統現代化（Legacy 救援）

情境：一個運行多年、沒文件、沒測試、原開發者已離職的系統，業務還離不開它。

工作流程：

  1. 用提示詞 1 產出上手報告，用提示詞 18 建立 AGENTS.md
  2. 用提示詞 3 追蹤最關鍵的 2～3 條業務路徑，用提示詞 13 為它們補上特徵測試（characterization tests）——先鎖住現有行為
  3. 用提示詞 2 盤點技術債並排序
  4. 依提示詞 7 分批升級過時依賴，每批一個 PR
  5. 最後用提示詞 6 重構核心模組，測試護航

結果：「沒人敢動」的系統變成有測試、有文件、可持續維護的系統——這正是方法 3 可以販售的完整服務。

案例 2：7 天做出一個 SaaS

情境：你有一個利基工具的點子和一週時間。

工作流程：

  1. 第 1 天：用提示詞 10 做出核心功能原型，親自試用、砍掉不必要的功能
  2. 第 2～3 天：用提示詞 9 補齊帳號、付款、資料持久化（每個功能一張工單）
  3. 第 4 天：用提示詞 13 補測試、提示詞 15 做安全健檢
  4. 第 5 天：架好 CI 與部署流程，開啟 GitHub 自動審查
  5. 第 6 天：用提示詞 20 做上線總檢查
  6. 第 7 天：上線，用提示詞 12 建一條使用數據的日報管線

結果：一週從點子到可收費的產品，且維護基礎（測試、CI、審查）從第一天就到位。

案例 3：團隊級審查與維護自動化

情境：5～15 人的工程團隊，PR 審查大排長龍，維護工作永遠排不進衝刺。

工作流程：

  1. 制定團隊的 code_review.md，從各儲存庫的 AGENTS.md 引用
  2. 開啟 GitHub 整合，前兩週採「僅標記」模式建立信任
  3. 把 CI 紅燈分流（提示詞 17）與每週依賴更新（提示詞 7）用 codex exec 排程化
  4. 在 Slack 工程頻道建立 @Codex 派工慣例與訊息格式
  5. 每月回顧：代理審查的準確率、自動 PR 的合併率，據此調整規範

結果：人類審查專注在設計決策，例行維護有排程保證，回報到修復的週期以小時計。

第 8 章・附錄

常見問題

Codex、Claude Code、Cursor、Copilot 怎麼選？

它們的定位不同：Copilot 核心是編輯器內補全與輔助；Cursor 是 AI 優先的編輯器體驗；Codex 和 Claude Code 是同一類——完整任務的工程代理，差別主要在生態（前者綁 ChatGPT／GitHub 整合，後者綁 Claude 生態）與模型特性。務實的做法：都在你的真實程式碼庫上試同一張工單，比 PR 品質，不比行銷數字。很多團隊同時用兩家的代理，依任務特性分流。

該選哪個模型？

用官方目前的預設建議即可（2026 年中為 GPT-5.5）。專用 Codex 型號的世代交替很快（5-Codex → 5.1 → 5.3 → 併入旗艦），追型號不如把 AGENTS.md 和工單品質顧好——那才是結果差異的主因。

我的訂閱額度能跑多少任務？

依方案與任務複雜度而定：Plus（$20）適合每天派幾張工單的個人使用；重度使用者選 Pro 5×（$100）或 Pro 20×（$200）；團隊用 Business 加額度。大型重構這類長任務消耗快，先跑價值最高的。

我的程式碼會被拿去訓練嗎？

商業方案（Business／Enterprise）預設不用你的資料訓練；個人方案請檢查資料控制設定中的訓練選項。企業導入前把資料處理政策讀一遍——這是採購流程該做的事，不是可選項。

沙箱安全嗎？讓它自動執行指令會不會出事？

本機沙箱預設限制網路與寫入範圍，雲端任務在隔離容器中執行。真正的風險控制在你手上：許可模式依專案風險調整（技巧 11），危險操作（部署、刪資料、動正式環境）永遠保留人工確認。

需要會寫程式才能用嗎？

雲端任務讓非工程師也能用一段描述得到可運行的原型（提示詞 10）。但要「持續交付可靠的軟體」，你至少需要具備驗收能力——看得懂它做了什麼、判斷得了對不對。這也是為什麼模組 6 說：你賣的是驗收的判斷力。

什麼時候用本機 CLI、什麼時候用雲端任務？

情境建議介面
需要即時來回、看著每個 diffIDE 擴充功能
本機環境相依、需要腳本化或接 CICodex CLI
獨立的大任務、可背景跑、平行派工雲端任務

它開的 PR 我全部都要細看嗎？

初期是的——這是建立信任的成本。之後讓分層把關接手：CI 擋機械性問題、代理審查（模組 5）擋邏輯問題、你只看設計決策與業務正確性。永遠抽查，永遠不要零審查合併。

怎麼知道我的任務適不適合交給 Codex？

三個檢查點：1）能不能寫成一張有驗收標準的工單？2）成果能不能用測試或實際執行驗證？3）規模是不是「一個函式以上、一次重寫以下」？三個都是，就派。
